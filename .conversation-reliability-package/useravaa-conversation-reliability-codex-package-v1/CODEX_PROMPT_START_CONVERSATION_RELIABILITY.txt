Feature:
Conversation Request Reliability System for Useravaa MVP.

Goal:
Prevent dead conversation requests, remove infinite pending states, preserve requester momentum, and make every conversation request time-bounded, stateful, and clearly actionable.

Scope for this phase:

1. Conversation request state model

Implement or update the request status model:
- pending_provider_response
- times_proposed
- pending_payment
- confirmed
- rejected
- expired
- cancelled

State rules:
- New request starts as pending_provider_response.
- Provider can propose times or reject.
- Provider must respond within 24 hours.
- If Provider does not respond within 24 hours, system expires the request.
- Provider must propose at least 3 times.
- After times are proposed, Requester has 48 hours to select one time.
- If Requester does not select within 48 hours, system expires the request.
- Payment is only available after Requester selects a proposed time.
- After successful payment, status becomes confirmed.
- No request can remain indefinitely pending.

- date required
- time required

7. Time selection and payment gating

Requester can select exactly one proposed time.

Rules:
- select-time allowed only when status = times_proposed
- after selection, status becomes pending_payment
- payment disabled before time selection
- payment enabled only when status = pending_payment
- after mock successful payment, status becomes confirmed

Error copy:

8. Expiration logic

Implement mock/system expiration behavior.

For MVP/prototype:
- add deterministic helper functions or fixture actions to simulate expiration
- if current time exceeds providerResponseDeadlineAt and status is pending_provider_response, status becomes expired
- if current time exceeds requesterSelectionDeadlineAt and status is times_proposed, status becomes expired

If background jobs are in scope for this codebase, model these jobs:
- expire pending provider requests every 15 minutes
- expire proposed times every 15 minutes
- conversation reminder every 15 minutes
- expiration warning every 1 hour

Do not connect real cron infrastructure unless already available.
Keep jobs mockable/testable.
